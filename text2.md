第1章 序言

1.1 设计背景

在高校资源循环利用与数字化校园建设的背景下，传统校园二手交易存在信息分散、交易效率低、缺乏统一审核与监管、安全保障不足等问题。随着移动互联网与微信生态的普及，以轻量化小程序为载体、搭配规范化管理后台的线上交易模式成为解决校园二手交易痛点的有效方案。本文设计并实现一套前后端分离的校园二手交易平台，系统采用Spring Boot后端服务 + Vue 3管理后台 + 微信小程序用户端的三层架构，依托MySQL数据库实现数据持久化，通过Redis提升访问效率，基于JWT完成双端身份认证。平台分为管理端与用户端，管理端支持管理员登录、数据统计仪表盘、商品审核、用户管理、订单管理、举报处理、商品分类与平台公告管理等运营功能；小程序用户端支持微信一键登录、商品浏览、分类筛选、搜索、发布、收藏、留言、订单交易、商品举报等全流程操作。本文完成了系统需求分析、总体架构设计、功能模块划分、数据库设计、核心代码实现与系统测试，最终实现一套稳定可用、操作便捷、监管规范的校园二手交易系统。该系统有效打通校园二手物品供需信息，提升交易效率，保障交易安全，满足学生轻量化使用与平台方统一运营管理的双重需求，具备较强的实用性与可扩展性。

关键词：校园二手交易；Spring Boot；Vue 3；微信小程序；JWT认证；前后端分离

1.2 设计意义

随着移动互联网与数字化校园建设的不断推进，线上化、规范化、高效化已成为校园生活服务的主流方向。开发并应用校园二手交易平台，具有显著的理论价值与现实意义。

从学生使用层面来看，本平台让闲置物品交易摆脱时间与空间限制，学生足不出校即可快速发布、浏览、搜索、购买所需物品，大幅降低交易成本、提升交易效率，真正实现"方便、快捷、安全"的二手交易体验，有效满足日常学习与生活需求。

从资源利用层面来看，平台推动校园闲置物品循环流转，减少物品浪费与重复购置，践行绿色低碳、节能环保的理念，对建设节约型校园、提升校园资源利用率具有重要的促进作用。

从校园管理层面来看，系统具备完整的审核、监管、数据统计功能，可有效规范交易行为、过滤违规信息、处理交易纠纷，保障校园网络环境安全有序，提升校园服务数字化与精细化管理水平。

1.3 设计要求

本项目旨在为高校师生搭建一个安全、规范、便捷、轻量化的校园二手交易平台，通过完整的用户端与管理后台体系，实现二手物品从发布、审核、展示、交易到监管、统计的全流程线上化运作。

系统整体分为用户交互与运营管理两大核心模块：

用户端采用微信小程序实现，遵循"即用即走"原则，界面简洁清晰、操作流程简短，充分利用手机移动端的便捷性，实现商品浏览、搜索、发布、收藏、下单、留言、举报等常用功能，确保学生能够快速上手、流畅使用。

管理后台采用Web管理系统实现，具备数据可视化、权限控制、统一审核等能力，支持商品审核、用户管理、订单管理、举报处理、分类维护、公告发布等运营功能，保证平台交易有序、内容合规、可查可管。

数据层面选用MySQL + Redis进行数据存储与缓存加速，系统稳定可靠、易于维护；接口采用RESTful规范与JWT认证，保障接口安全与权限隔离。系统整体要求结构清晰、扩展性强，能够满足校园场景下的高并发访问与稳定运行，符合高校数字化、绿色化、规范化的建设要求。

2.4 前后端分离架构

系统采用标准前后端分离模式：

后端仅提供RESTful API，负责业务逻辑、数据持久化、权限校验、异常处理；

管理后台（Vue 3）与小程序（原生）作为独立前端，通过Axios/wx.request调用接口；

接口统一返回标准格式：code、message、data、timestamp；

认证通过请求头Authorization: Bearer传递，前后端完全解耦。

第3章 关键问题及分析

在设计与实现过程中，对系统安全性和稳定性要求较高，系统采用UI显示和数据处理分离的方式来解决这一问题。

3.1 研究设计中要解决的问题

在研究设计中，还存在着诸多有待解决的问题与困难，会用到许多以前未学过的难点去学习与攻破。现将研究设计中要解决的问题总结如下：

3.1.1 数据表间联系

由于系统中管理的数据较多，包括用户、商品、订单、分类、举报、公告等多个实体，表间的联系建立至关重要。如果表间的关联建立不够完善，当程序运行时会出现数据不一致、查询效率低下等问题，程序不便于维护。本系统通过合理的外键关联与索引设计，确保数据完整性与查询性能。

3.1.2 系统安全性

由于本系统涉及用户交易与个人信息安全，需要严格的用户身份验证与权限控制。系统采用JWT令牌认证机制，用户通过微信登录或管理员账号密码登录后获取Token，后续请求通过Token验证身份。对于敏感操作（如订单创建、商品发布）需要进行身份校验，确保操作用户的身份合法性。

3.1.3 订单状态流转

订单状态管理是交易系统的核心。系统设计了完整的状态流转机制：待支付 -> 已支付 -> 卖家已确认/买家已确认 -> 已完成，同时支持取消与退款。状态流转需要保证原子性与一致性，避免出现订单状态混乱的情况。

3.1.4 商品审核流程

为保障平台内容合规，系统设计了商品审核流程：用户发布商品 -> 待审核状态 -> 管理员审核 -> 通过/驳回。驳回的商品用户可修改后重新提交，审核结果与驳回原因需要清晰记录，便于用户了解审核状态。

第6章 系统实现及测试

6.1 系统主要模块的实现

6.1.1 后端服务搭建

后端服务采用Spring Boot框架搭建，主要实现以下功能模块：统一响应结构、全局异常处理、JWT认证拦截、数据库连接池配置、Redis缓存配置、MyBatis-Plus ORM配置。

统一响应结构采用ApiResponse封装所有接口返回值，包含code（状态码）、message（消息）、data（数据）、timestamp（时间戳）四个字段。

全局异常处理通过@ControllerAdvice注解实现，统一捕获并处理业务异常（BusinessException）与系统异常，返回友好的错误信息。

JWT认证通过Filter拦截请求，验证Token有效性，从Token中解析用户ID与角色信息，存入请求上下文供后续使用。

数据库连接池采用Druid，配置初始化连接数、最大连接数、连接超时时间、连接检测等参数，同时启用监控页面。

系统配置包括文件上传大小限制、Redis连接配置、JWT密钥与过期时间、微信登录配置等。

接口文档通过Knife4j实现，访问/doc.html可查看完整的API文档与在线调试界面。

6.1.2 用户端微信登录模块

微信登录是小程序用户的主要入口，实现流程如下：

（1）用户点击登录按钮，调用wx.login()获取code

（2）将code发送到后端/user/auth/wx-login接口

（3）后端调用微信接口code2Session，传入code、appid、appsecret

（4）微信返回openid与session_key，后端根据openid查询或创建用户

（5）后端生成JWT Token，返回给小程序

（6）小程序将Token存储到本地，后续请求携带Token

登录成功后，用户信息（ID、昵称、头像）显示在个人中心，用户可使用完整功能。

6.1.3 商品浏览与搜索模块

商品浏览模块实现商品列表展示、分类筛选、排序功能。

列表查询接口支持分页、分类ID筛选、关键词搜索、排序方式（最新、价格）参数。查询结果包含商品基本信息（ID、标题、价格、封面图、浏览数、收藏数）、卖家信息（昵称、头像）、分类信息。

商品详情接口返回完整商品信息，包括多张商品图片、商品描述、发布时间、卖家详细信息（昵称、头像、发布商品数）、收藏状态（当前用户是否收藏）、留言列表。

搜索功能支持按商品标题和描述关键词匹配，使用MySQL的LIKE查询，结合分类筛选和排序条件。

6.1.4 商品发布与审核模块

商品发布流程：用户填写商品标题、描述、价格，选择分类，上传商品图片，提交审核。

图片上传采用本地存储方案，后端保存图片到服务器uploads目录，返回访问URL。多个图片时存储到goods_image表，关联商品ID。

商品提交后状态变为PENDING，在审核列表中展示。管理员可查看商品详情，选择通过或驳回。驳回时填写驳回原因，用户可在商品详情查看驳回原因，修改后重新提交。

审核通过后商品状态变为APPROVED，在用户端商品列表展示。用户可随时将商品下架（下架后状态变为OFFLINE），下架后仍可重新上架（上架后需重新审核）。

6.1.5 订单交易模块

订单交易是平台的核心功能，实现完整的交易流程：

（1）买家选择商品，点击购买，创建订单，状态为PENDING_PAYMENT

（2）买家进行模拟支付，调用支付接口，订单状态变为PAID

（3）卖家收到订单通知，确认发货，订单流转到SELLER_CONFIRMED

（4）买家确认收货，或系统自动确认，交易完成，订单状态变为COMPLETED

（5）买家或卖家可取消未支付的订单，订单状态变为CANCELLED

订单创建时校验商品状态、库存、买家是否为自己、订单金额等信息。支付状态与订单状态独立管理，支持交易异常情况的处理。

6.1.6 管理后台仪表盘模块

管理后台仪表盘展示平台运营数据，帮助管理员了解平台状况。

统计卡片展示五项核心指标：用户总数、商品总数、待审核商品数、订单总数、待处理举报数。

近7日趋势图采用ECharts折线图，展示每日新增用户、新增商品、新增订单的趋势变化，数据来源于各表的每日统计。

商品分类分布采用ECharts饼图，展示各类目的商品数量占比，帮助了解平台商品结构。

统计数据通过Redis缓存30秒，减少数据库查询压力。

6.1.7 商品审核管理模块

商品审核是平台内容质量保障的关键环节。

审核列表默认展示待审核商品，按提交时间倒序排列，支持按分类筛选。列表展示商品基本信息、卖家信息、提交时间、操作按钮。

点击审核按钮进入审核详情页，展示完整商品信息（图片、描述、价格）、卖家信息、历史审核记录（如有）。

审核操作包括通过和驳回。驳回时必须填写驳回原因，系统将原因通知用户。审核结果记录到goods_audit表，包含审核ID、商品ID、管理员ID、审核结果、驳回原因、审核时间。

6.1.8 用户与订单管理模块

用户管理展示平台所有用户列表，包含用户ID、昵称、头像、手机号、状态、注册时间、操作。管理员可禁用账号，被禁用用户无法登录和使用大部分功能。

订单管理展示平台所有交易订单，按创建时间倒序排列。列表展示订单编号、商品标题、买家、卖家、金额、订单状态、支付状态、操作。可按订单状态、支付状态筛选。

点击订单编号可查看订单详情，包括商品信息、买卖双方信息、订单时间线、操作记录。

第7章 总结
